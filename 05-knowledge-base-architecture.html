How to read this

1 The agent starts at Layer 5 (the specific use case) and reads up through every layer above it. A Nova error message gets: error message rules + Nova channel rules + tone matrix + servicing principles + Capitec voice + brand foundation — all at once.
2 Upper layers override lower layers when there is a conflict — except Layer 0 (Legal). Legal and regulatory requirements override all other layers, including Brand Foundation and Bold Simplicity. This is a hard rule, not a tone judgment. Legal copy must never be simplified into non-compliance. Within Layers 1–5, Brand Foundation wins.
3 Nova (Layer 4) has a split context — client-facing screens follow App-like rules, SE-facing screens follow a different register. This must be explicit in the Nova MD file or the agent will average the two and be wrong for both.
4 Escalation rules (Layer 5) flag output that must not ship without a human writer reviewing it — sensitive moments, legal copy, new product launches. These are non-negotiable checkpoints regardless of output quality.
5 The knowledge base is only as good as the content in each layer. Layers 1–3 are foundational and stable. Layers 4–5 need regular writer-led maintenance as products and channels evolve.
6 Servicing Principles (Layer 2): The CX head has flagged these as required inputs. An MD file exists — retrieve, review, and validate before adding to the KB. Owner: Louise. Without this layer, the agent can produce technically correct copy that violates Capitec's service commitments.
7 Segments and personas are out of T1 scope. Channels cannot yet accommodate that level of personalisation. Layer 4 focuses on UI constraints and contextual rules instead — character limits, real estate, time on task, SE presence. This is more reliable and easier to codify than audience personalisation. Segments and personas can be added to the architecture in T2 once the base agent is proven.
12 Layer 3 is intentionally simplified for T1. The full tone matrix — product × journey stage × emotional moment — has been replaced with four emotional states (error/failure, success, instructional, sensitive/urgent). This covers 90% of what designers encounter and is far faster to build and more reliably applied by the agent. Good vs bad examples are embedded directly into each Layer 5 use case file rather than maintained as a separate document. Both decisions can be revisited in T2 once the agent is live and generating real output data that reveals where more nuance is needed.
8 Legal vs Bold Simplicity — this conflict is real and must be named. Legal copy often conflicts directly with Bold Simplicity. The agent cannot resolve this on its own — it will default to simplifying, which risks non-compliance. Layer 0 is the explicit resolution: legal wins, every time. Locked disclosures must appear verbatim. The agent flags any output containing locked legal text for writer + legal review before shipping.
9 What Nathan already has (confirmed 2026-03-30): Nathan has a working Copilot Studio agent that brand copywriters are currently using. His KB already contains: Layer 0 (legal & compliance), Layer 1 (brand), and Layer 2 (servicing — needs sense-checking for scope and accuracy). He has also uploaded UX writing principles. He is actively investigating Copilot Studio's native trigger capabilities for bringing writers into the loop automatically — this aligns directly with the escalation requirement. Priority: review and validate his existing layers before building anything new. Do not duplicate work.
9 How this runs in Copilot Studio (T1 technical approach): Layers 0–3 plus escalation rules are always loaded via the system prompt. Layer 4 (channel) and Layer 5 (use case) files live in SharePoint and are retrieved automatically — designers never paste MD files. When a designer opens the agent, a built-in briefing template guides them: channel, copy type, what to communicate, constraints. The agent only generates after collecting all four. When the agent outputs REVIEW REQUIRED, a Power Automate flow automatically messages the relevant writer in Teams and confirms to the designer. Leon owns the technical configuration.
11 Two tools, one knowledge base. The Copilot Studio writing agent (generate + review) and the Figma plugin (layered assessment: UI checks → UX checks → copy checks) both reference the same SharePoint MD files. When a writer updates an MD file, both tools update automatically. The Figma plugin is Phase 2 scope. Its copy check layer calls the same underlying agent as the writing tool — just in assessment mode, not generation mode. One KB, two interfaces, one escalation flow.
10 T1 channels: App, Nova, ATM. T2: POS, Salesforce. App and Nova are the primary T1 build. ATM is included in T1 because its constraints are clear and distinct (very limited real estate, short TOT, time pressure). POS and Salesforce Layer 4 files do not exist yet — they must not be referenced by the agent until they do.
